# ADR-0004: Два кода доступа для customer-портала

## Status
Accepted.

## Context
Гарантийный талон должен:
1. Иметь публичную проверку подлинности (любой человек по QR-коду может убедиться, что документ настоящий).
2. Защищать ПДн клиента в публичной проекции (152-ФЗ).
3. Давать клиенту возможность оставить отзыв так, чтобы посторонние не могли спамить от его имени.

Один общий код не удовлетворяет одновременно (1) и (3): если кодом для отзыва будет тот же, что в QR на распечатке, любой увидевший PDF сможет писать отзывы за клиента.

## Decision
Хранить **два** независимых кода в `installations`:
- `verification_code` (12 hex, 48 бит) — публичный, попадает в QR.
- `access_token` (16 hex, 64 бита) — личный, печатается отдельной строкой в PDF.

Оба сравниваются через `hash_equals` (constant-time).

`customer.php` определяет `accessLevel`:
- сначала проверяет `access_token` → если совпал, **personal** (всё открыто, можно оставить отзыв);
- иначе проверяет `verification_code` → если совпал, **public** (ПДн маскируются, отзыв оставить нельзя);
- иначе — none, форма поиска.

QR в PDF кодирует **только** публичный URL. Личная ссылка печатается рядом текстом — клиент сам выбирает, давать её кому-то или нет.

## Consequences
- Распечатанный QR на стенке у клиента можно показывать гостям — они получат только public-просмотр.
- Личный URL клиент может дать только тем, кому доверяет — отзывы привязаны к этому коду.
- В БД у каждого монтажа два кода с гарантированно разной длиной (12 vs 16 hex), коллизия исключена.
- При утере PDF клиентом — он может попросить переотправить личную ссылку через монтажника (внутренний admin её видит в `installation_edit.php`).

## Alternatives considered
- **Только один код с уровнями доступа через query-параметр.** Отвергнуто — клиент может случайно расшарить full-access URL.
- **OTP по SMS / email.** Отвергнуто для MVP — нет SMTP / SMS-провайдера в скоупе.
